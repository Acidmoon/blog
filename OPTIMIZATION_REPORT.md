# 博客项目优化审查与修复报告

审查与实施日期：2026-07-11

## 结论

本轮按第一性原理处理了审查中列出的数据一致性、输入边界、请求成本和部署韧性问题。核心状态变化现在遵循“完整提交或完整回滚”的原则；派生数据可重建，历史正文可离线恢复，首页不再在请求路径等待第三方服务。

没有发现需要立即停机处理的 P0 问题。剩余风险主要是生产环境尚未进行真实 Docker 构建、反向代理和多进程压测验证，不能用本地单机测试替代。

## 已完成修复

| 审查项 | 已实施保障 |
| --- | --- |
| SQLite 与点赞并发 | 连接统一启用 WAL、外键和 5 秒 busy timeout；点赞在短事务内线性化，锁竞争返回 503 而非唯一索引 500。启动时只读检查关系漂移，仅在需要时重建。 |
| 标签迁移与 URL | 旧标签按确定规则保留有效项、跳过无效项并重建陈旧关系；所有标签与分页链接已编码，覆盖 `R&D`、`C#`、`%` 和空格。 |
| 聊天事务 | 模型成功后，会话、双消息与初始标题在一个事务内落库；故障注入覆盖中段和末段回滚。消息历史默认分页，并支持加载更早消息。 |
| 全文搜索 | 新增 SQLite FTS5 索引；文章创建、编辑、删除在原事务内同步索引。公开搜索先分页匹配，再读取当前结果页正文；支持 `python -m services.search --rebuild`。 |
| 活动热力图 | 新增不可变 `article_activity_events`；创建、编辑和发布操作同事务记录活动与字数变化。热力图使用事件聚合、索引友好的时间范围，并处理公元 1 年和 9999 年边界。 |
| 首页一言 | 页面即时返回当天/陈旧缓存或本地回退；远端刷新移到跨进程锁保护的后台线程，限制上游响应类型、大小和字段长度。 |
| 正文版本生命周期 | 新增离线 dry-run 和 `--offline --apply` 隔离工具，将旧版本移到带 manifest 的恢复目录，不允许在线删除。 |
| 错误与片段契约 | HTML/API 错误响应稳定且不泄露异常详情；首页 AJAX 和热力图片段会检查 HTTP 状态、恢复 loading 状态并显示错误。 |
| 认证与存储维护 | CSRF、可信代理、上传流式校验与默认凭据保护继续生效；过期访客令牌按限频周期清理。 |
| 部署 | 新增只读 `/healthz`，Docker 探针改为该端点；默认端口绑定回环地址，可显式覆盖；部署脚本记录旧 revision/image，并在健康失败时只回滚目标服务。 |

- 本机没有 Docker CLI；Docker build、Compose 实际运行、反向代理信任链和真实多 worker 竞争仍需在受控环境验证。

## 运维边界

1. 生产部署仍需显式授权，且不得触碰 `data/`、`.env`、容器或远端服务之外的资源。
2. `article_maintenance --apply --offline` 只可在文章写入已停止的维护窗口运行。
3. `BLOG_BIND_ADDRESS` 默认是 `127.0.0.1`；经反向代理发布时，应同时配置实际直连代理的 `TRUSTED_PROXY_CIDRS`。
4. FTS 和活动事件是派生/历史数据；若维护或迁移异常，可先使用 dry-run、索引重建和恢复批次 manifest 进行核验，避免直接删除文件或数据库记录。
